# 兵种与部队编制模块设计 v1

## 1. 模块定位

`兵种与部队编制模块` 负责定义部队实体与兵种模板，承接城市资源并生成可作战部队，不直接结算战斗伤害与地形空间规则。

目标是形成稳定规则链路：  
`城市资源/库存 -> 编制校验 -> 部队组装 -> 行动状态流转 -> 输出给战斗/补给/地图模块消费`

### 1.1 顶层变量（v1 默认值）
- `ARMS_TYPES = [CAVALRY, SPEARMAN, HALBERD_SHIELD, ARCHER, CATAPULT, TRANSPORT]`  
  v1 兵种枚举。
- `UNIT_STACK_LIMIT_PER_TILE = 1`  
  部队占格上限（与地图模块一致，不可叠堆）。
- `UNIT_EXECUTION_STATE = [UNEXECUTED, EXECUTED]`  
  部队执行状态二分口径。
- `MAX_ACTION_COUNT_PER_TURN = 1`  
  单位每回合仅可执行 1 次操作（普通进攻/战法/策略/补给）。
- `ACTION_SET_REMAINING_MOVE_POINT_TO_ZERO = true`  
  执行任一操作后行动力归 0。
- `NO_UNIT_ON_CITY_TILES_HARD = true`  
  城市 7 格内永无部队，入城在编制层拆散入库。
- `CITY_COMPOSE_HARD_CAP_ENABLED = true`  
  编制出征必须同时满足城市资源硬约束与城市可编制能力硬约束。
- `UNIT_MONEY_CAP_FIXED = 10000`  
  部队携带钱上限固定 10000。
- `UNIT_FOOD_CAP_TURN_SPAN = 20`  
  部队携带粮上限按“当前兵力可支撑 20 回合”计算。
- `NEW_UNIT_DEFAULT_EXECUTION_STATE = UNEXECUTED`  
  新建部队默认状态为 `UNEXECUTED`。
- `NEW_UNIT_CONTROL_SCOPE = CURRENT_FACTION_TURN_ONLY`  
  新建部队仅在当前势力回合可控。
- `CITY_ENTRY_REQUIRE_ADJACENT_TILE = true`  
  入城拆解仅允许由城市相邻格（环距离 1）触发。
- `CITY_ENTRY_REQUIRE_REMAINING_MOVE_POINT_GT_ZERO = true`  
  入城拆解触发前，部队必须 `remainingMovePoint > 0`。
- `DISBAND_CITY_OVERFLOW_POLICY = DISCARD_OVERFLOW`  
  入城拆解回库时，超过城市可接收上限的资源直接消失，不保留、不转存。
- `CITY_ACTION_LIMIT_FOR_RECOMPOSE = UNLIMITED`  
  城市内兵种重组不受“城市行动次数”限制。
- `CATAPULT_COMPOSE_MODE = ONE_CATAPULT_PER_UNIT`  
  投石车编制按“1 个投石车兵器 = 1 支投石车部队”。

## 2. 模块边界（做什么 / 不做什么）

## 2.1 负责内容
- 管理兵种基础模板（攻防、机动、可用战法标签、基础粮耗）。
- 处理兵源 + 兵器 + 资源到部队的组装、重组与出征。
- 处理部队“入城拆解”与资源回流城市库存。
- 维护部队运行态（兵力、兵器、携带资源、战法点、执行状态）。
- 输出战斗/补给/地图所需的标准化部队实体快照。

## 2.2 不负责内容
- 不负责地块可达、路径消耗、视野遮挡等空间判定（由地形与地图模块负责）。
- 不负责命中、伤害、伤员生成与攻城破城结算（由战斗与攻城模块负责）。
- 不负责补给距离判定与补给结算（由补给与后勤模块负责）。
- 不负责兵种×地形交叉修正表维护（由规则框架与数值治理模块负责）。

## 3. 对外输入输出定义（模块级）

## 3.1 输入（Input）

### A. 城市资源输入（来自城市发展模块）
- `cityTroopSource`: 可用兵源。
- `cityWoundedTroopSource`: 城市伤员池。
- `cityWeaponStockByType`: 各兵种对应兵器库存。
- `cityCatapultStock`: 可用投石车器械数量。
- `cityCarryableResources`: 可编制时可分配的钱粮等资源。
- `maxComposableForces`: 当前城市可编制能力硬上限。
- `maxRecruitableTroopSource`: 当前城市可转化兵源硬上限。

### B. 编制请求输入
- `factionId`: 势力标识。
- `cityId`: 发起编制城市。
- `composeRequestList`: 目标兵种、目标兵力、目标兵器、目标资源负载。
- `composeRuleVersion`: 编制规则版本。

### C. 运行态输入
- `turnIndex`: 当前回合序号。
- `unitExecutionStateSnapshot`: 当前势力各单位执行状态快照。
- `unitActionEvent`: 单位本回合移动/操作事件流。
- `currentTurnFactionId`: 当前回合势力标识（用于新建部队可控性校验）。
- `foodConsumptionPerTurnByArmsType`: 各兵种单回合粮耗参数（用于粮上限计算）。

### D. 参数表输入（来自规则框架与数值治理模块）
- `armsTemplateTable`: 兵种基础属性模板。
- `composeCapRuleTable`: 编制上限、最小起编量、资源占用约束。
- `resourceCarryCapRuleTable`: 部队携带资源上限规则。
- `catapultComposeRuleTable`: 投石车编制与兵源绑定规则。
- `statModifierRuleTable`: 数值修正规则表（平值/百分比/乘区/截断口径）。

### E. 修正层上下文输入（来自外部模块）
- `generalModifierSnapshotByUnitId`: 武将修正快照（后续接入）。
- `terrainModifierSnapshotByUnitId`: 地形修正快照（由地形与地图模块提供标签映射）。
- `statusModifierSnapshotByUnitId`: 状态修正快照（士气、断粮、控制等）。
- `cityAcceptableCapacitySnapshot`: 城市可接收容量快照（入城回库截断口径，按兵源/伤员/兵器/粮/钱分项）。

## 3.2 输出（Output）

### A. 编制执行输出
- `isComposeExecuted`: 是否编制成功。
- `composeFailReason`: 失败原因（资源不足、规则冲突、超上限）。
- `createdUnitList`: 新建部队实体列表。
- `cityResourceDelta`: 城市库存变化明细。

### B. 部队实体输出（供外部模块消费）
- `unitId / factionId / armsType`
- `troopSource / weaponAmount / catapultAmount`
- `woundedTroopSource`
- `atk / def / move / int / build`
- `finalStatSnapshot`:
  - `finalAtk / finalDef / finalMove / finalInt / finalBuild`
- `finalCombatDetailSnapshot`（结算层消费，不在基础展示层直出）:
  - `finalMeleeAtk / finalRangedAtk / finalMeleeDef / finalRangedDef`
- `food / money / carryResourceSnapshot`
- `resourceCapSnapshot`:
  - `troopSourceCap`: 出城时兵源值（编制完成瞬间冻结）。
  - `weaponCap`: 使用 `troopSourceCap` 值（`weaponCap = troopSourceCap`）。
  - `foodCap`: `(troopSource + weaponAmount) * singleTurnFoodCost * 20`。
  - `moneyCap`: 固定 `10000`。
- `tacticPoint / tacticTagSet`
- `executionState / remainingMovePoint`
- `isControllableInCurrentTurn`: 是否可被当前势力在本回合直接控制。

### C. 入城拆解输出
- `isDisbandExecuted`: 是否入城拆解成功。
- `disbandResourceReturn`: 拆解回库明细（兵源、伤员、兵器、粮草、钱）。
- `disbandOverflowDiscard`: 超出城市可接收上限而被丢弃的资源明细（含伤员）。
- `unitRemoved`: 是否移除大地图单位实体。

## 4. 核心流程（v1）

## 4.1 部队编制流程
1. 接收编制请求并读取城市库存快照。
2. 执行城市硬约束校验：
   - 校验城市库存资源是否足够；
   - 校验 `maxComposableForces` 与 `maxRecruitableTroopSource` 是否允许本次编制；
   - 任一失败即拒绝编制并返回明确原因码。
3. 校验兵种模板、兵器匹配、最小起编量、资源上限。
4. 扣减城市资源并创建单位实体。
5. 初始化新建部队：
   - `executionState = UNEXECUTED`（新建即生效）；
   - `isControllableInCurrentTurn = true`，但仅对 `currentTurnFactionId` 生效。
   - `woundedTroopSource = 0`（新编制出征部队伤员初始化为 0）。
6. 执行数值修正层计算并写入 `finalStatSnapshot`（若无修正输入则等于基础值）。
7. 以 `finalMove` 刷新 `remainingMovePoint`（新编制出征时立即生效）。
8. 写入 `resourceCapSnapshot`：
   - 兵力上限（兵源+兵器）按出城时状态冻结；
   - `weaponCap` 固定使用 `troopSourceCap`；
   - 粮上限为当前兵力 20 回合总消耗；
   - 钱上限固定 10000。
9. 输出编制日志与可出征部队列表。

## 4.2 单位行动状态流转
1. 单位被选中并移动时，仅减少 `remainingMovePoint`，不自动置为已执行。
2. 单位执行一次操作（普通进攻/战法/策略/补给）后：
   - `executionState -> EXECUTED`
   - `remainingMovePoint -> 0`
3. 势力回合结束时，当前势力全部可控单位统一置为 `EXECUTED`。
4. 势力回合开始时，当前势力全部可控单位统一重置为 `UNEXECUTED`，并以部队移动值刷新行动力：
   - 先执行修正层，得到 `finalMove`；
   - `remainingMovePoint = finalMove`。

## 4.3 入城拆解流程
1. 触发前置：己方部队先执行常规移动，抵达目标己方城市的任意相邻格（环距离 1）。
2. 触发方式：部队仍被选中时，点击目标城市（交互风格与“选中部队后点击目标地块”一致）。
3. 状态校验：仅当 `remainingMovePoint > 0` 且 `executionState = UNEXECUTED` 时允许触发入城拆解。
4. 校验目标城市归属与可接收状态；若失败，返回失败原因并保持部队原地。
5. 计算回库增量并按城市可接收上限截断；溢出部分直接丢弃并记录 `disbandOverflowDiscard`。
6. 将截断后的兵源、伤员、兵器、资源转回城市库存（伤员进入城市伤员池，不可直接操作）。
7. 移除地图单位实体，确保“城市 7 格内永无部队”。
8. 输出拆解日志供城市发展与战斗统计消费。

## 5. 部队实体属性与公式（冻结口径）

## 5.1 实体字段分层
- **标识层**：`unitId`、`factionId`、`armsType`、`cityId`（来源城市，可选）。
- **兵力层**：`troopSource`、`woundedTroopSource`、`weaponAmount`、`catapultAmount`。
- **基础数值层**：`atk`、`def`、`move`、`int`、`build`。
- **资源层**：`food`、`money`、`carryResourceSnapshot`。
- **上限层**：`resourceCapSnapshot`（`troopSourceCap`、`weaponCap`、`foodCap`、`moneyCap`）。
- **战斗行为层**：`tacticPoint`、`tacticTagSet`、`executionState`、`remainingMovePoint`。
- **控制层**：`isControllableInCurrentTurn`。

## 5.2 基础数值定义（编制后生效）
- `troopSource`：部队人数，近似等价于当前血量；当 `troopSource <= 0` 时部队溃败并移除战场实体。
- `atk`：攻击值，影响普通攻击/部分战法的伤害基础项。
- `def`：防御值，影响受击减伤与部分战法抗性基础项。
- `move`：移动值，影响回合开始时该部队 `remainingMovePoint` 初始化值。
- `int`：智力值，影响策略类效果成功率，以及抵抗敌方策略的能力。
- `build`：建设值，影响部队执行建设类操作（设施建造/修复等）的进度速度。
- 上述数值在编制完成时由兵种模板写入，作为“基础数值”；后续可被武将系统二次修正。

### 5.2.1 兵种基础排序口径（当前设想冻结）
- 攻击排序：`骑兵 > 枪兵 > 弓兵 > 戟盾兵 > 投石车`。
- 防御排序：`戟盾兵 > 枪兵 = 投石车 > 骑兵 > 弓兵`。
- 该排序用于兵种模板配置与平衡校准，具体数值由 `armsTemplateTable` 落地。
- 若后续数值治理调整模板绝对值，需保持相对排序不被破坏（除非新版本显式改版）。

### 5.2.2 隐藏结算维度（不在基础展示层直出）
- 攻击在结算层区分为：`meleeAtk`（近战攻击）、`rangedAtk`（远程攻击）。
- 防御在结算层区分为：`meleeDef`（近战防御）、`rangedDef`（远程防御）。
- 基础展示层仍只展示 `atk / def` 汇总值，避免 UI 复杂度上升。
- 近/远程攻防由基础攻防和系数派生，不单独作为基础模板展示字段：
  - `meleeAtk = atk * meleeAtkCoeff`
  - `rangedAtk = atk * rangedAtkCoeff`
  - `meleeDef = def * meleeDefCoeff`
  - `rangedDef = def * rangedDefCoeff`
- 战斗结算时由战斗模块依据攻击类型读取对应维度；若缺少系数配置，默认系数为 `1.0`。

## 5.3 关键字段公式
- `weaponCap = troopSourceCap`。
- `moneyCap = 10000`。
- `foodCap = (troopSource + weaponAmount) * singleTurnFoodCost * 20`。  
  其中 `singleTurnFoodCost` 来自 `foodConsumptionPerTurnByArmsType[armsType]`。
- `executionState` 初始化：`UNEXECUTED`。
- `isControllableInCurrentTurn = (unit.factionId == currentTurnFactionId)`。
- `remainingMovePoint(onCompose) = finalMove`（新编制出征时刷新）。
- `remainingMovePoint(turnStart) = finalMove`（回合开始刷新）。
- `isRouted = (troopSource <= 0)`。
- `woundedTroopSource(onCompose) = 0`。

## 5.4 数值修正层（新增）
- 目标：在不改写编制层基础模板的前提下，统一汇总“武将/状态/地形/战法”等修正，产出可结算的 `finalStatSnapshot`。
- 分层口径：
  - `baseStatSnapshot`：编制模块写入的基础值（`atk / def / move / int / build`）。
  - `baseCombatDetailSnapshot`：结算侧细分值（`meleeAtk / rangedAtk / meleeDef / rangedDef`，由 `atk / def` 与系数派生）。
  - `modifierSnapshot`：各来源修正项集合（平值、百分比、乘区标签）。
  - `finalStatSnapshot`：用于战斗/移动/策略/建设结算的最终值。
- 推荐计算顺序（v1）：
  1. `baseValue = baseStatSnapshot[stat]`
  2. `flatAdded = sum(modifierFlat[stat])`
  3. `percentAdded = sum(modifierPercent[stat])`
  4. `finalValue = floor((baseValue + flatAdded) * (1 + percentAdded))`
  5. `finalValue` 再经过硬上限/下限截断（由数值治理表提供）。
- 修正层输出：
  - `finalAtk / finalDef / finalMove / finalInt / finalBuild`
  - `finalMeleeAtk / finalRangedAtk / finalMeleeDef / finalRangedDef`
  - `finalMove` 在回合开始用于刷新 `remainingMovePoint`
  - `modifierBreakdownLog`（用于战报与回放）
- 边界约束：修正层只产出最终数值，不处理命中/伤害/成功率的最终结算（仍由战斗或对应模块执行）。

## 5.5 武将系统预留修正口径
- 当前阶段：先使用兵种基础值，不接入武将修正。
- 后续接入武将系统后，接入到数值修正层统一处理。
- 武将修正建议最少覆盖字段：`atk / def / move / int / build`。

## 5.6 编制时写入规则
1. 编制成功后写入 `troopSource / weaponAmount / catapultAmount / food / money`。
2. 同步写入 `atk / def / move / int / build`（来源于 `armsTemplateTable`）。
3. 立即冻结 `resourceCapSnapshot`。
4. 若输入资源超过上限，按上限截断；截断量进入编制日志。
5. 新建单位状态固定为 `UNEXECUTED`，并仅当前势力回合可控。
6. 城市内兵种重组不受城市行动次数限制，可在满足资源条件下连续执行。

## 5.7 投石车编制冻结规则
- 每消耗 `1` 个投石车兵器，创建 `1` 支投石车部队（`catapultAmount = 1`）。
- 每支投石车部队可绑定“任意兵源”作为编制兵力输入（只需满足最小存活兵源阈值，若项目未设阈值则默认 `> 0`）。
- 投石车编制成功后，兵源与投石车兵器同步扣减；部队其余资源按常规编制流程写入。
- 同一城市回合内可连续编制多支投石车部队，仅受城市库存与可编制能力硬约束限制。

## 5.8 入城拆解与回库公式
- `resourceReturnApplied = min(unitResource, cityAcceptableCapacity)`。
- `resourceOverflowDiscard = unitResource - resourceReturnApplied`。
- 对每类资源（兵源/伤员/兵器/粮/钱）独立计算上述两式并分别记账。
- 当 `resourceOverflowDiscard > 0` 时，溢出资源直接消失，不进入任何缓冲池。
